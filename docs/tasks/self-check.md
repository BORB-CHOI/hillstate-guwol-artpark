# 작업: 3분 내 집 마련 준비도 체크

## 배경

대출 총량관리와 정책모기지 금리 상승으로, 고객이 홍보관에 오기 전에 "내가 지금 살 수 있는 상황인가"를 먼저 확인하려는 흐름이 강해졌다. 분양정보 열람보다 자가 진단이 앞에 온다.

담당자(박진 과장) 요구사항 정리:

- 정확한 진단일 필요 없다. 어차피 방문해서 상담한다.
- 걸러야 할 핵심은 두 가지뿐이다. 무주택자인지, 신용에 문제가 없는지.
- 금액, 소득, 부채 액수 등 숫자 입력은 받지 않는다.
- 응답 기반 결과 화면이 이 기능의 메인이다.

## 범위

1. 조건형 설문 페이지
2. 응답 기반 결과 화면
3. 진단 결과 DB 저장

범위 밖: 대출 가능액 계산, 금융 심사 연동, 가격 노출, 로그인.

## 반드시 지킬 원칙

- **가격 노출 금지.** 분양가, 계약금, 중도금, 대출 한도, 월 상환액 등 어떤 금액도 화면에 띄우지 않는다. 결과 문구에도 숫자 금액을 쓰지 않는다.
- **확정 표현 금지.** "대출 가능합니다", "승인됩니다" 같은 단정 표현을 쓰지 않는다. 최종 안내는 항상 홍보관 방문과 금융기관 심사로 넘긴다.
- 결과 화면 하단 고정 문구:
  > 정확한 공급조건과 금융 가능 여부는 홍보관 방문과 금융기관 심사를 통해 확인해 주세요.
- 문체는 CLAUDE.md 한글 문체 규칙을 따른다. em dash와 가운뎃점을 장식으로 쓰지 않고, 번역투를 피한다.

## 1. 설문

경로: `/check` (신규 라우트)

7문항, 한 화면에 한 문항씩 진행하고 상단에 진행 표시를 둔다. 모두 선택형이며 자유 입력은 없다.

| # | 질문 | 선택지 |
|---|---|---|
| 1 | 현재 주택을 보유하고 계신가요 | 무주택 / 1주택 / 2주택 이상 |
| 2 | 가족 구성은 어떻게 되시나요 | 1인 / 신혼부부 또는 예비부부 / 자녀 있는 가구 / 부모님 포함 3세대 |
| 3 | 지금 어떤 형태로 거주하고 계신가요 | 전세 / 월세 / 자가 / 부모님 댁 |
| 4 | 어떤 타입을 보고 계신가요 | 59㎡ / 74㎡ / 84㎡ / 아직 고민 중 |
| 5 | 입주는 언제쯤 생각하고 계신가요 | 최대한 빨리 / 1~2년 뒤 / 입주 시점에 맞춰 / 아직 미정 |
| 6 | 기존 대출이 있으신가요 | 없음 / 신용대출 있음 / 주택담보대출 있음 / 여러 건 있음 |
| 7 | 금융 상담을 받아보시겠어요 | 예 / 아니요 |

문항 개수는 7개로 고정한다. 임의로 추가하거나 빼지 않는다.

마지막 단계에서 이름과 연락처, 개인정보 수집 이용 동의를 받는다. 검증 규칙은 `app/api/reservations/route.js`의 `validPhone`과 동일하게 쓴다. 허니팟(`company`) 숨김 필드도 동일하게 넣는다.

## 2. 결과 화면

분기 기준은 1번(주택 보유)과 6번(기존 대출) 두 축만 쓴다. 나머지 응답은 결과 문구를 개인화하는 데만 쓴다.

**A. 준비 양호** — 무주택 + 대출 없음 또는 신용대출만
> 무주택 상태에서 준비하고 계셔서 청약과 자금 계획을 세우기 좋은 조건입니다. 홍보관에서 타입별 조건과 일정을 함께 확인해 보세요.

**B. 확인 필요** — 무주택 + 주담대 보유 또는 여러 건
> 기존 대출 상환 계획에 따라 준비 방법이 달라집니다. 방문 시 대출 현황을 함께 정리해 드립니다.

**C. 상담 권장** — 1주택 이상
> 보유 주택 처분 계획에 따라 조건이 달라집니다. 담당자와 개별 상담이 필요한 유형입니다.

각 유형마다 "방문 전 확인할 항목"을 체크리스트로 보여준다. 항목은 서류와 확인 사항만 담고 금액은 넣지 않는다. 예: 무주택 확인 자료, 세대 구성 기준, 기존 대출 잔여 조건, 입주 시점 자금 일정.

7번에서 "예"를 고른 경우 결과 하단에 방문예약 CTA를 강조해 붙이고, "아니요"인 경우 일반 CTA만 노출한다.

## 3. 저장

`lib/db.js`에 `createSelfCheck`, `listSelfChecks`, `deleteSelfCheck`를 추가한다. 기존 `reservations` 구현을 그대로 따르되 리스트 키만 `self_checks`로 분리한다. Redis 우선, 없으면 로컬 파일 폴백하는 구조를 유지한다.

저장 레코드:

```
id, name, phone, answers(7문항 코드값), result_type(A|B|C),
wants_consult, source, utm_*, created_at
```

원본 선택지 문구 대신 코드값으로 저장한다. 나중에 문구를 바꿔도 집계가 깨지지 않게 한다.

API: `app/api/self-check/route.js`

- `POST` 접수. 서버에서 답변 값을 화이트리스트로 검증하고, 결과 유형도 서버에서 다시 계산한다. 클라이언트가 보낸 `result_type`은 신뢰하지 않는다.
- `GET`, `DELETE` 관리자용. `ADMIN_TOKEN` 검증 방식은 예약 API와 동일하게 맞춘다.

알림은 `lib/notify.js`의 텔레그램 발송을 재사용한다. 방문예약 알림과 구분되도록 제목에 진단 접수임을 표시하고, 결과 유형과 상담 희망 여부를 함께 보낸다. 알림 실패가 접수 실패로 이어지지 않게 하는 처리도 예약 API와 동일하게 둔다.

관리자 화면(`app/admin/page.js`)에 진단 결과 탭을 추가한다. 목록은 접수 시각, 이름, 연락처, 결과 유형, 상담 희망 여부를 보여준다.

## 4. UI

- 진입 애니메이션은 `components/Reveal.js`를 쓴다. 새 라이브러리를 추가하지 않는다.
- 설문 안내 문구는 가운데 정렬, 큰 글자로 둔다. 좌측 정렬 소형 텍스트로 만들지 않는다.
- 선택지는 큼직한 카드형 버튼으로 만들고 hover, focus, 선택 상태를 각각 다르게 준다.
- 랜딩 페이지 안에서 진단으로 들어가는 진입 배너를 방문예약 폼 위쪽에 하나 둔다.
